# Правила построения OData-запросов к 1С:Предприятие

Документ описывает правила формирования URL для запросов к стандартному OData-интерфейсу 1С:Предприятие.
Правила извлечены из обработки «Конструктор OData» (`docs/конструктор_odata.bsl`).

---

## 1. Базовая структура URL

```
{АдресСервиса}/odata/standard.odata/{ИмяРесурса}[?{Параметры}]
```

**Пример:**

```
http://127.0.0.1/MyBase/odata/standard.odata/Catalog_Сотрудники?$top=10&$format=json
```

### Компоненты

| Часть | Описание | Пример |
|-------|----------|--------|
| `{АдресСервиса}` | Адрес веб-публикации ИБ | `http://127.0.0.1/MyBase` |
| `/odata/standard.odata/` | Путь к стандартному OData-сервису | — |
| `{ИмяРесурса}` | Имя набора сущностей (EntitySet) | `Catalog_Сотрудники` |
| `?{Параметры}` | Опциональные параметры запроса | `$top=10&$filter=...` |

> **Важно:** Если адрес сервиса не заканчивается на `/`, он добавляется автоматически.

## 2. Имена ресурсов (EntitySet) по типам объектов 1С

### 2.1. Таблица соответствия префиксов

Имя ресурса формируется как **`Префикс_ИмяОбъектаМетаданных`** (в 1С-кодировке).

| Тип объекта 1С | Префикс OData | Пример имени ресурса |
|----------------|---------------|----------------------|
| Справочник | `Catalog_` | `Catalog_Сотрудники` |
| Документ | `Document_` | `Document_ЗаявкаНаОтпуск` |
| Журнал документов | `DocumentJournal_` | `DocumentJournal_Общий` |
| План видов характеристик | `ChartOfCharacteristicTypes_` | `ChartOfCharacteristicTypes_ВидыСубконто` |
| План счетов | `ChartOfAccounts_` | `ChartOfAccounts_Основной` |
| План видов расчёта | `ChartOfCalculationTypes_` | `ChartOfCalculationTypes_Начисления` |
| План обмена | `ExchangePlan_` | `ExchangePlan_ОбменС_sites` |
| Константа | `Constant_` | `Constant_АдресПубликации` |
| Регистр сведений (записи) | `InformationRegister_{Имя}_RecordType` | `InformationRegister_ЦеныНоменклатуры_RecordType` |
| Регистр сведений (срез последних) | `InformationRegister_{Имя}_СрезПоследних` | `InformationRegister_ЦеныНоменклатуры_СрезПоследних` |
| Регистр накопления (записи) | `AccumulationRegister_{Имя}_RecordType` | `AccumulationRegister_ОстаткиТоваров_RecordType` |
| Регистр расчёта (записи) | `CalculationRegister_{Имя}_RecordType` | `CalculationRegister_Начисления_RecordType` |
| Регистр бухгалтерии (записи) | `AccountingRegister_{Имя}_RecordType` | `AccountingRegister_Хозрасчетный_RecordType` |
| Бизнес-процесс | `BusinessProcess_` | `BusinessProcess_СогласованиеЗаявки` |
| Задача | `Task_` | `Task_ЗадачаИсполнителя` |

### 2.2. Табличные части

Табличные части объектов (справочников, документов, бизнес-процессов, задач) получают имя ресурса по правилу:

```
{ПрефиксРодителя}_{ИмяОбъекта}_{ИмяТабличнойЧасти}
```

| Объект | Табличная часть | Имя ресурса |
|--------|-----------------|-------------|
| Справочник `Сотрудники` | `ТрудоваяДеятельность` | `Catalog_Сотрудники_ТрудоваяДеятельность` |
| Документ `ЗаявкаНаОтпуск` | `ДниОтпуска` | `Document_ЗаявкаНаОтпуск_ДниОтпуска` |

## 3. Стандартные свойства (поля) по типам объектов

### 3.1. Справочники

| Свойство OData | Тип | Описание |
|----------------|-----|----------|
| `Ref_Key` | GUID | Уникальный идентификатор (ссылка) |
| `DeletionMark` | Boolean | Пометка удаления |
| `Predefined` | Boolean | Предопределённый элемент |
| `IsFolder` | Boolean | Является ли папкой (иерархический справочник) |
| `Parent_Key` | GUID | Ссылка на родителя (иерархия) |
| `Code` | String | Код элемента |
| `Description` | String | Наименование |

Плюс все **реквизиты** объекта из метаданных.

### 3.2. Документы

| Свойство OData | Тип | Описание |
|----------------|-----|----------|
| `Ref_Key` | GUID | Уникальный идентификатор (ссылка) |
| `DeletionMark` | Boolean | Пометка удаления |
| `Date` | DateTime | Дата документа |
| `Number` | String | Номер документа |
| `Posted` | Boolean | Признак проведения |

Плюс все **реквизиты** объекта из метаданных.

### 3.3. Регистры сведений

#### Записи (`_RecordType`)

| Свойство OData | Тип | Описание |
|----------------|-----|----------|
| `RecordKey` | String | Ключ записи |
| `Period` | DateTime | Период (только если регистр периодический) |

Плюс **измерения** и **ресурсы** из метаданных.

#### Срез последних (`_СрезПоследних`)

| Свойство OData | Тип | Описание |
|----------------|-----|----------|
| `Period` | DateTime | Период |

Плюс **измерения** и **ресурсы** из метаданных.

### 3.4. Регистры накопления / расчёта / бухгалтерии (записи)

Стандартные поля аналогичны регистрам сведений (`RecordKey`, `Period`), плюс измерения, ресурсы и реквизиты из метаданных.

### 3.5. Табличные части

| Свойство OData | Тип | Описание |
|----------------|-----|----------|
| `LineNumber` | Number | Номер строки |

Плюс все **реквизиты** табличной части из метаданных.

### 3.6. Остальные типы объектов

Для всех прочих типов (планы видов характеристик, планы счетов, планы видов расчёта, планы обмена, бизнес-процессы, задачи) доступны как минимум:

| Свойство OData | Тип | Описание |
|----------------|-----|----------|
| `Ref_Key` | GUID | Уникальный идентификатор |
| `DeletionMark` | Boolean | Пометка удаления |

Плюс все **реквизиты** объекта из метаданных.

## 4. Суффиксы для типов реквизитов

Для каждого реквизита объекта могут генерироваться дополнительные свойства в зависимости от типа данных.

### 4.1. Ссылочный тип (`_Key`)

Если реквизит имеет ссылочный тип (ссылка на справочник, документ и т.д.), автоматически создаётся свойство с суффиксом **`_Key`**, возвращающее UUID связанного объекта.

| Реквизит 1С | Тип | Свойство OData | Значение |
|-------------|-----|----------------|----------|
| `ФизическоеЛицо` | `CatalogRef.ФизическиеЛица` | `ФизическоеЛицо` | Представление (навигационное свойство) |
| — | — | `ФизическоеЛицо_Key` | `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` |

**Пример использования в фильтре:**

```
$filter=ФизическоеЛицо_Key eq guid'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
```

### 4.2. Составной тип (`_Type`)

Если реквизит имеет составной тип (несколько возможных типов значений), автоматически создаётся свойство с суффиксом **`_Type`**, указывающее фактический тип значения.

| Реквизит 1С | Тип | Свойство OData | Значение |
|-------------|-----|----------------|----------|
| `Владелец` | `CatalogRef.Контрагенты, CatalogRef.ФизическиеЛица` | `Владелец` | Представление |
| — | — | `Владелец_Key` | UUID |
| — | — | `Владелец_Type` | Имя типа (например, `CatalogRef.Контрагенты`) |

> **Как определить тип:** Тип является ссылочным, если среди его возможных типов есть `СправочникСсылка.*` или `ДокументСсылка.*`. Тип является составным, если он содержит более одного возможного типа (`Типы().Количество() > 1`).

## 5. Параметры запроса

Все параметры — опциональны. При указании нескольких параметров они объединяются через `&`.

### 5.1. `$top` — Ограничение количества записей

```
$top=<N>
```

Возвращает первые N записей.

**Пример:** `$top=10`

### 5.2. `$skip` — Пропуск записей

```
$skip=<N>
```

Пропускает первые N записей. Используется для постраничной навигации совместно с `$top`.

**Пример:** `$skip=20&$top=10` — записи с 21-й по 30-ю.

### 5.3. `$filter` — Фильтрация

```
$filter=<Условие>
```

Поддерживаемые операторы сравнения:

| Оператор | Описание | Пример |
|----------|----------|--------|
| `eq` | Равно | `Code eq '001'` |
| `ne` | Не равно | `DeletionMark ne true` |
| `gt` | Больше | `Date gt datetime'2024-01-01T00:00:00'` |
| `ge` | Больше или равно | `Code ge '005'` |
| `lt` | Меньше | `Date lt datetime'2024-12-31T23:59:59'` |
| `le` | Меньше или равно | `Code le '010'` |

Строковые функции:

| Функция | Описание | Синтаксис | Пример |
|---------|----------|-----------|--------|
| `substringof` | Содержит подстроку | `substringof('строка', Поле)` | `substringof('Иванов', Description)` |
| `startswith` | Начинается с | `startswith(Поле, 'строка')` | `startswith(Code, '00')` |
| `endswith` | Заканчивается на | `endswith(Поле, 'строка')` | `endswith(Description, 'ов')` |

> **Внимание:** `substringof` принимает аргументы в порядке `(подстрока, поле)`, а не `(поле, подстрока)`.

Логические связки:

| Связка | Описание | Пример |
|--------|----------|--------|
| `and` | Логическое И | `DeletionMark eq false and Code eq '001'` |
| `or` | Логическое ИЛИ | `Code eq '001' or Code eq '002'` |

**Формат записи фильтра:**

```
Поле Оператор Значение
```

Несколько условий соединяются через `and` или `or`:

```
$filter=DeletionMark eq false and substringof('Иванов', Description)
```

### 5.4. `$orderby` — Сортировка

```
$orderby=Поле1 [asc|desc][, Поле2 [asc|desc], ...]
```

| Направление | Описание |
|-------------|----------|
| `asc` | По возрастанию (по умолчанию) |
| `desc` | По убыванию |

Если направление не указано, по умолчанию используется **`asc`**.

**Примеры:**

```
$orderby=Description asc
$orderby=Date desc, Number asc
```

### 5.5. `$select` — Выбор полей

```
$select=Поле1,Поле2,Поле3
```

Возвращает только указанные поля. Если параметр не задан — возвращаются все поля.

**Пример:**

```
$select=Ref_Key,Description,Code
```

### 5.6. `$expand` — Раскрытие навигационных свойств

```
$expand=Поле1,Поле2
```

Разворачивает навигационные свойства (ссылочные реквизиты), подставляя вместо ключей полные объекты.

**Пример:**

```
$expand=ФизическоеЛицо
```

### 5.7. `$format` — Формат ответа

```
$format=json|atom
```

| Значение | Описание |
|----------|----------|
| `json` | Данные в формате JSON |
| `atom` | Данные в формате Atom/XML (по умолчанию) |

## 6. Полный пример URL

### Справочник: выбрать 10 не удалённых сотрудников с наименованием, содержащим «Иванов»

```
http://127.0.0.1/MyBase/odata/standard.odata/Catalog_Сотрудники?$top=10&$filter=DeletionMark eq false and substringof('Иванов', Description)&$select=Ref_Key,Code,Description&$orderby=Description asc&$format=json
```

### Документ: все проведённые документы за 2024 год

```
http://127.0.0.1/MyBase/odata/standard.odata/Document_ЗаявкаНаОтпуск?$filter=Posted eq true and Date gt datetime'2024-01-01T00:00:00' and Date lt datetime'2025-01-01T00:00:00'&$format=json
```

### Регистр сведений: срез последних цен

```
http://127.0.0.1/MyBase/odata/standard.odata/InformationRegister_ЦеныНоменклатуры_СрезПоследних?$filter=Period eq datetime'2024-06-01T00:00:00'&$format=json
```

### Табличная часть документа

```
http://127.0.0.1/MyBase/odata/standard.odata/Document_ЗаявкаНаОтпуск_ДниОтпуска?$top=50&$format=json
```

## 7. Алгоритм формирования запроса

На основе кода конструктора OData, запрос формируется в следующем порядке:

1. **Выбрать адрес сервиса** (`{АдресСервиса}`) — из настроек или константы `АдресПубликацииИнформационнойБазыВЛокальнойСети`.
2. **Выбрать ресурс** — имя ресурса из дерева метаданных (`{ИмяРесурса}`).
3. **Сформировать базовый URL:** `{АдресСервиса}/odata/standard.odata/{ИмяРесурса}`.
4. **Добавить параметры** в порядке:
   - `$top` (если задано значение > 0)
   - `$skip` (если задано значение > 0)
   - `$filter` (если есть хотя бы одно условие фильтрации)
   - `$orderby` (если задана хотя бы одна сортировка)
   - `$select` (если выбран хотя бы один флаг у поля)
   - `$expand` (если выбран хотя бы один флаг раскрытия)
   - `$format` (если выбран формат `json`; `atom` — по умолчанию, не добавляется)
5. **Соединить параметры** через `&` и добавить `?` перед первым параметром.

### Правила формирования строки фильтра

```
Условие = Поле + " " + Оператор + " " + Значение
```

Несколько условий:

```
СтрокаФильтра = Условие1 [ " and " | " or " Условие2 [ " and " | " or " ... ] ]
```

> Первое условие добавляется без связки. Каждое последующее — с `and` или `or`.

### Правила формирования строки сортировки

```
СтрокаСорт = Поле1 [ " desc" ] [, Поле2 [ " desc" ], ...]
```

> Направление `asc` не указывается явно (используется по умолчанию). Только `desc` добавляется к имени поля.
